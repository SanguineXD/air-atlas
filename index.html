ESP32 Air Quality Station — Build Guide

BME280 · PMS7003 · MQ-7 (MOSFET cycling) · MQ-135 — hover underlined terms for quick definitions

Before you start — what is all this?

The ESP32 is a low-cost 32-bit dual-core microcontroller with integrated Wi-Fi and Bluetooth. It executes firmware written in C++, interfaces directly with sensors via its GPIO pins, and is programmed through the Arduino IDE on a host computer.

Sensors are electronic components that convert physical or chemical phenomena — temperature, humidity, particulate concentration, gas levels — into electrical signals. The ESP32 samples these signals via I²C, UART, or analogue-to-digital conversion, and processes the resulting data.

ThingSpeak is an IoT cloud platform that ingests time-series sensor data via HTTP. The ESP32 transmits readings over Wi-Fi, where they are stored in a channel and made available through a public REST API.

The Air Atlas dashboard queries ThingSpeak's API and renders the data as interactive visualisations in the browser. No coding knowledge is required to operate the system once configured.

About the components: The BME280 is an integrated sensor chip that provides temperature, humidity, and pressure over a single I²C bus — three measurements in one component. The 2N7000 is a small N-Channel MOSFET that switches the MQ-7 heater on and off via PWM from the ESP32. The 1kΩ resistor between the GPIO and Gate limits current and protects the ESP32.

Understanding voltage levels

Voltage levels describe how devices encode digital signals. The ESP32's GPIO pins use 3.3V logic, meaning a HIGH signal is 3.3V. Sensors using 5V logic (such as MQ-7 and MQ-135) cannot be connected directly to ESP32 input pins, as 5V exceeds the ESP32's tolerance and will damage the chip.

Voltage divider for MQ sensors

The MQ-7 and MQ-135 are powered by 5V and their AOUT pins can output the full 0–5V range. Most ESP32 ADC pins tolerate only 0–3.3V, so a voltage divider is required to scale the signal down safely.

Formula: Vout = Vin × R2 ÷ (R1 + R2)
With R1 = 10kΩ and R2 = 20kΩ: Vout = 5V × 20 ÷ 30 = 3.33V ✓

Build this divider twice — once for MQ-7 AOUT → GPIO35, once for MQ-135 AOUT → GPIO33.

Note on module variants: Some MQ breakout boards include built-in level shifters that limit AOUT to 3.3V, but this varies by manufacturer and is not always documented. Using a voltage divider is the safe default in all cases — it does no harm if the module already limits voltage, and it protects the ESP32 if it does not.

Why the MQ-7 needs voltage cycling

The MQ-7 has a metal oxide heater element inside. The datasheet specifies alternating phases:

Phase 1 — CLEANING
5V for 60 seconds
Burns off impurities. Do NOT take readings.
→
Phase 2 — MEASUREMENT
1.4V for 90 seconds
Sensor is now sensitive to CO. Read in last 2-10s.
→
Repeat
150s full cycle
Continuous accurate sensing

The PWM math — 7.8% duty cycle

Instead of a separate 1.4V power supply, the ESP32 "pulses" the 5V supply so quickly that the heater feels like it's at 1.4V.

Important: Heating power is proportional to the square of voltage (P ∝ V²). So the duty cycle is NOT 1.4/5 = 28%. It's:

Duty Cycle = (Target ÷ Source)² = (1.4 ÷ 5)² = 0.078 ≈ 7.8%

In ESP32 code (8-bit PWM resolution), this is PWM value 20 out of 255.
PhasePWM ValueEffective VoltageDurationAction
Cleaning255 (100%)5.0V60 secondsHeat — no readings
Measurement20 (7.8%)1.4V effective90 secondsRead in last 2-10 seconds

The 2N2222 transistor circuit

The ESP32 can't directly switch the MQ-7's heater current. A 2N2222 NPN transistor acts as the switch, controlled by GPIO32 through a mandatory 1kΩ base resistor.

NEVER connect the 2N2222 to the ESP32 without the 1kΩ resistor. Direct connection will draw too much current from the GPIO and can damage the ESP32.

ESP32 firmware (replace the 3 placeholders)

#include <Wire.h>
#include <Adafruit_BME280.h>
#include <HardwareSerial.h>
#include <WiFi.h>
#include <HTTPClient.h>

const char* ssid     = "YOUR_WIFI_SSID";
const char* password = "YOUR_WIFI_PASSWORD";
const char* apiKey   = "YOUR_THINGSPEAK_WRITE_KEY";  // from ThingSpeak tab
const char* tsURL    = "http://api.thingspeak.com/update";

#define PMS_RX 16  #define PMS_TX 17
#define MQ7_ADC 34  #define MQ135_ADC 35  #define MQ7_GATE 26
#define PWM_CH 0  #define PWM_HZ 1000  #define PWM_BITS 8
#define DUTY_HIGH 255  #define DUTY_LOW 72   // 28% ≈ 1.4V

Adafruit_BME280 bme;
HardwareSerial pmsSerial(2);
bool mq7High=true; uint32_t mq7Start=0;
float mq7Last=0; bool mq7Ready=false;

struct Pms{uint16_t pm1,pm25,pm10;};
bool readPMS(Pms &d){
  if(pmsSerial.available()<32||pmsSerial.read()!=0x42||pmsSerial.read()!=0x4D)return false;
  uint8_t b[30]; pmsSerial.readBytes(b,30);
  d.pm25=(b[4]<<8)|b[5]; d.pm10=(b[6]<<8)|b[7]; return true;
}
int avgADC(int pin,int n=20){long s=0;for(int i=0;i<n;i++){s+=analogRead(pin);delay(3);}return s/n;}
void updateMQ7(){
  uint32_t e=millis()-mq7Start;
  if(mq7High){if(e>=60000){ledcWrite(PWM_CH,DUTY_LOW);mq7High=false;mq7Start=millis();mq7Ready=false;}}
  else{if(e>=60000&&!mq7Ready){mq7Last=avgADC(MQ7_ADC);mq7Ready=true;}
       if(e>=90000){ledcWrite(PWM_CH,DUTY_HIGH);mq7High=true;mq7Start=millis();}}
}
void setup(){
  Serial.begin(115200);
  pmsSerial.begin(9600,SERIAL_8N1,PMS_RX,PMS_TX);
  Wire.begin(21,22);
  if(!bme.begin(0x76)&&!bme.begin(0x77)){Serial.println("BME280 not found!");while(1);}
  ledcSetup(PWM_CH,PWM_HZ,PWM_BITS);
  ledcAttachPin(MQ7_GATE,PWM_CH);
  ledcWrite(PWM_CH,DUTY_HIGH); mq7Start=millis();
  WiFi.begin(ssid,password);
  for(int i=0;i<30&&WiFi.status()!=WL_CONNECTED;i++){delay(500);Serial.print(".");}
}
void loop(){
  updateMQ7();
  float temp=bme.readTemperature(),hum=bme.readHumidity(),pres=bme.readPressure()/100.0f;
  int mq135=avgADC(MQ135_ADC);
  Pms pms={0,0,0};
  uint32_t t0=millis(); while(millis()-t0<1500)if(readPMS(pms))break;
  float co=mq7Ready?mq7Last:-1;
  // Send to ThingSpeak every 15s (free tier minimum)
  static uint32_t lastSend=0;
  if(millis()-lastSend>=15000&&WiFi.status()==WL_CONNECTED){
    HTTPClient http;
    String url=String(tsURL)
      +"?api_key="+apiKey
      +"&field1="+String(temp,1)+"&field2="+String(hum,1)
      +"&field3="+String(pres,1)+"&field4="+String(co,0)
      +"&field5="+String(mq135)+"&field6="+String(pms.pm25)
      +"&field7="+String(pms.pm10)
      +"&field8="+String(pms.pm1);
    http.begin(url);
    Serial.println(http.GET()==200?"TS: OK":"TS: error");
    http.end(); lastSend=millis();
  }
  delay(500);
}

Data flow — second by second

  1. ESP32 wakes up → 60 seconds heating MQ-7 at 5V → 90 seconds at 1.4V (reading CO in last 30s)
  2. Reads BME280 (temp/hum/pres), PMS7003 (PM1.0/PM2.5/PM10), MQ-135 (general AQ)
  3. Sends single HTTP GET to api.thingspeak.com/update with 8 fields
  4. ThingSpeak stores the row with a UTC timestamp and entry ID
  5. Dashboard visitor opens Air Atlas → fetches last 8000 entries via JSON API
  6. Every 3 mins after that, dashboard silently refreshes without clearing the screen
  7. Apps Script trigger fires every 30 mins → loops through all 10 channels
  8. For each channel: fetches last 10 entries, checks if timestamp already exists in Sheet
  9. If new, appends row with IST-converted timestamp, station name, city, and all 8 sensor values
  10. Sheet now has permanent record. Available forever for capstone analysis.

Step 3 — Fetching live data from ThingSpeak

The function below fetches the last N results for any channel as JSON:

// Fetch last `results` readings from a ThingSpeak channel
async function fetchChannel(channelId, readKey, results = 500) {
  const url = `https://api.thingspeak.com/channels/${channelId}/feeds.json`
            + `?api_key=${readKey}&results=${results}`;
  const res  = await fetch(url);
  const json = await res.json();

  // json.feeds is an array of readings:
  // { created_at: "2026-04-27T09:14:32Z", field1: "31.2", field6: "42.1", ... }
  return json.feeds.map(f => ({
    ts:    new Date(f.created_at).getTime(),
    temp:  parseFloat(f.field1),
    hum:   parseFloat(f.field2),
    pres:  parseFloat(f.field3),
    co:    parseFloat(f.field4),
    mq135: parseFloat(f.field5),
    pm25:  parseFloat(f.field6),
    pm10:  parseFloat(f.field7),
    pm1:   parseFloat(f.field8)
  })).filter(r => !isNaN(r.temp));
}

// Fetch ALL 10 channels in parallel (much faster than sequential)
async function fetchAllStations() {
  const results = await Promise.all(
    TS_CHANNELS.map(ch => fetchChannel(ch.channelId, ch.readKey, 500))
  );
  // results[0] = array of readings for station 0, etc.
  return results;
}

// Auto-refresh every 30 seconds
async function startLiveMode() {
  await fetchAllStations();        // initial load
  setInterval(fetchAllStations, 30000); // refresh every 30s
}

Step-by-step assembly

Always power off (unplug USB) before adding or moving wires.
1
Place ESP32 on the 830-point breadboard, straddling the centre gap so each pin sits on its own row.
2
Place 2N7000 MOSFET on the breadboard with the flat face toward you. Pins from left to right: Source · Gate · Drain.
3
Wire the 2N7000: Source → GND rail · Gate → 1kΩ resistor → ESP32 GPIO32 · Drain → MQ-7 GND pin (later in step 7).
4
Wire BME280: VCC → 3V3 · GND → GND · SDA → GPIO21 · SCL → GPIO22.
5
Wire PMS7003: Pin 1 (VCC) → VIN (5V) · Pin 2 (GND) → GND · Pin 9 (TX) → GPIO16 · Pin 8 (RX) → GPIO17. (TX always crosses to RX.)
6
Build voltage divider A for MQ-7: R1 10kΩ + R2 20kΩ in series, junction → GPIO35.
7
Wire MQ-7: VCC → ESP32 5V/Vin pin · GND → 2N7000 Drain (NOT direct to ground rail) · AOUT → top of divider A.
8
Build voltage divider B for MQ-135 and wire MQ-135: VCC → 5V · GND → GND · AOUT → top of divider B → GPIO33.
9
Final checks:
✓ 1kΩ resistor between GPIO32 and 2N7000 Gate
✓ MQ-7's GND goes to MOSFET Drain, not ground rail
✓ TX/RX swap on PMS7003 (TX→GPIO16, RX→GPIO17)
✓ All other component GNDs share the common rail
10
Upload firmware: Board → "ESP32 Dev Module" · correct port · Serial Monitor at 115200. MQ sensors need 24–48h warm-up for calibrated readings.

Glossary

GPIO

The metal pins on your ESP32. Software sets them HIGH (3.3V) or LOW (0V). Max safe input: 3.3V — connecting 5V permanently damages the chip.

ADC — Analog-to-Digital Converter

Reads a voltage (0–3.3V) and converts it to a number 0–4095. MQ sensor readings arrive through these pins after voltage dividers scale them down.

PWM — Pulse Width Modulation

Simulates a lower voltage by rapidly switching ON/OFF. At 28% duty: 5V × 0.28 = 1.4V effective. Used to control the MQ-7 heater in the LOW phase.

Voltage divider

Two series resistors that split voltage proportionally. R1=10kΩ, R2=20kΩ → Vout = 5V × 20/30 = 3.33V. Protects ESP32 ADC from MQ sensors' 5V output.

MOSFET

An electronic switch controlled by voltage. GPIO26 drives the Gate, allowing current Drain→Source to switch the MQ-7 heater. IRLZ44N switches fully at 3.3V — safe for ESP32 direct drive.

I²C

Two-wire protocol: SDA (data) + SCL (clock). BME280 uses it. Multiple devices share the same two wires via unique addresses (0x76 or 0x77).

UART

Serial protocol: TX sends, RX receives. TX always crosses to the other device's RX. PMS7003 streams data at 9600 baud on GPIO16/17.

Flyback diode (1N4007)

Inductive loads generate reverse voltage spikes when switched off. The 1N4007 clamps these, protecting the MOSFET. Stripe (cathode) always toward +V.

Heater element

A resistive wire coil inside MQ sensors heats a metal oxide bead to ~300°C. Gas molecules change the bead's resistance — the AOUT pin reflects this as voltage.

AOUT — Analog Output

Continuous 0–5V pin on MQ modules. Higher gas = higher voltage. Must be divided to 0–3.3V before any ESP32 GPIO pin.

Voltage cycling (MQ-7)

5V for 60s clears the sensing surface. 1.4V for 90s lets CO bind — resistance is measured here. Without cycling, readings drift and saturate.

ThingSpeak Read API Key

A public key that lets anyone fetch your channel's data over HTTPS. Safe to share — it can only read, not write. The Write API Key must stay private on your device.
